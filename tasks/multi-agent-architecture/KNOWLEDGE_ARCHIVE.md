## 一、实验驱动方法论

### 1.1 不要让框架在数据之前

**原则**：让数据回答"该观测什么"，而不是预设框架再验证。

**反面**：S·λ·R·C → ρ → u → e 这个链本身就是先验——预设了"信息密度 = 四个因子相乘"，预设了"有用性 = 被引用次数"。

**正确做法**：
1. 先采集**纯行为数据**——发生了什么、引用了什么、压缩率多少、触发链是什么
2. 不做任何价值判断
3. 用无监督方法（变点检测、降维、图演化）让模式自己浮现
4. 模式浮现后，再根据涌现的发现构建模型

**具体模式**：
- **变点检测（PELT/CUSUM）**：在时间序列上自动发现"什么时候突变"，不预设衰减形状
- **PCA 降维**：多维行为向量降维到 2D/3D，看行为模式是否漂移、是否聚簇
- **图拓扑演化**：引用关系建图，观察聚类系数下降（联系瓦解）、连通分量增多（碎片化）

### 1.2 双轨并行作为科学保险

**原则**：先验框架和涌现式观测平行运行，互相校验。

**适用场景**：任何有强烈直觉但直觉可能偏差的研究问题。

**双轨对比**：

| 维度 | 先验轨道 | 涌现式轨道 |
|------|---------|-----------|
| 记录什么 | 框架定义的变量 | 原始行为事实 |
| 分析方式 | 监督学习（校正模型） | 无监督学习（模式发现） |
| 能验证什么 | 框架准不准 | 框架遗漏了什么 |
| 风险 | 框架本身就歪了 | 噪音中看不到信号 |

**交叉验证逻辑**：
- 两条轨道结论一致 → 置信度大幅提升
- 不一致 → 先验框架有盲区，值得深究
- 都不显著 → 可能问题本身不存在（本身也是科学发现）

### 1.3 假设树管理

**原则**：每个假设显式声明、追踪置信度、关联到数据来源。

**格式**：
```
| 假设 | 内容 | 置信度 | 关联轨道 | 验证方法 |
```

**关键假设类型**：
- H₀（零假设）：没有效应 → 实验设计的逻辑起点
- H₁（备择假设）：有效应 → 实验要找的东西
- Hδ（方法论假设）：测量方法本身是否可靠
- Hμ（交付物假设）：校正/干预是否有效

### 1.4 从实验到可信科学研究的四个阶段 🆕 v2.0

**这是本次更新最重要的方法论产出**。将"跑实验收集数据"系统地推进为"可发表、可复现的科学发现"需要经过四个层次：

| 层次 | 周期 | 核心活动 | DENSITY-001 当前状态 |
|------|------|---------|---------------------|
| **第一层：坚固实验基础** | 1-2周 | 数据累积至200+条，丰富特征维度，建立健康索引 | 100条验证残差，3个行为维度偏少 |
| **第二层：核心发现框架** | 2-4周 | 假设形式化 + 统计检验（t-test / bootstrap / Spearman rho） | 4项发现已识别，待统计检验 |
| **第三层：可复现科学命题** | 4-8周 | 提炼通用命题，跨对象泛化验证（DENSITY-002） | 未启动 |
| **第四层：成果产出** | 持续 | 正式报告 + 通用诊断器原型 + 可复现代码包 | DENSITY_001_PHASE1_REPORT.md 已完成 |

**关键洞察**：实验从"工具"到"科学"的跃迁点在于**命题的形式化和泛化验证**——一个观测对象的发现只能算工程报告，两个以上独立对象的交叉验证才能算科学。

### 1.5 统计检验的规范选择指南 🆕 v2.0

| 检验目的 | 应使用的方法 | DENSITY-001 匹配的假设 |
|---------|-------------|---------------------|
| 残差均值是否显著偏离零 | 单样本 t-test | Hδ |
| 变点的时间置信区间 | PELT + bootstrap | Hτ |
| 两个时间序列的关联强度 | Spearman rank correlation | Hγ |
| 模型跨时间泛化性能 | Time Series Cross-Validation (TS-CV) | Ridge v1 |

### 1.6 科学发现的可信度层级 🆕 v2.0

```
L1: "看起来好像" — 肉眼观察，无统计
L2: "统计显著" — t-test p<0.05, bootstrap CI
L3: "跨对象复现" — 在第二个独立 agent 上也成立
L4: "机制性理解" — 能解释为什么发生，能预测何时发生
```

DENSITY-001 当前处于 L1→L2 过渡期。4项发现已识别但尚未通过统计检验。

### 2.5 回归模型系数解释规范 🆕 v2.0

**当从黑箱模型提取可解释特征时**，系数应按下述方式解读：

| 系数符号 | 含义 | 实例 |
|---------|------|------|
| 正系数 | 特征被低估——该因子的实际贡献比模型估计的更多 | R=+0.675：相关性被低估 |
| 负系数 | 特征被高估——该因子被给予过高的先验权重 | ρ=-0.974：高密度被严重高估 |
| 近零系数 | 估计准确或数据不足 | S=-0.042：惊喜度估计相对准确 |
| 正截距 | 全局上修偏压——即使所有特征归零 | intercept=+0.507 |

## 三、跨机器文件传输

### 3.1 优先层

```
1. file_fetch / file_write（OpenClaw 内置）—— 有 agent 节点时优先
2. HTTP 临时服务（python -m http.server）—— 局域网 fallback
3. SMB 共享 —— 需两端 CIFS 服务就绪，不可靠
```

### 3.2 HTTP 临时传输模式

**适用**：两台 Windows 机器在同一局域网，SMB 不通。

**流程**：

**发送端（本机）**：
```powershell
# 1. 进入文件目录
cd target_directory

# 2. 起 HTTP 服务（后台）
Start-Process -NoNewWindow python -ArgumentList "-m","http.server","9999"

# 3. 添加入站防火墙规则（需管理员）
netsh advfirewall firewall add rule name="Temp HTTP 9999" dir=in action=allow protocol=TCP localport=9999
```

**接收端（对端）**：
```powershell
# 拉取文件
curl http://192.168.1.x:9999/template.html -o E:\path\template.html
```

**清理（发送端）**：
```powershell
# 关服务
Stop-Process -Name python -Force
# 删防火墙规则
netsh advfirewall firewall delete rule name="Temp HTTP 9999"
```

**教训**：
- 传输完必须立即清理（关服务 + 删防火墙规则），留临时端口是安全隐患
- 9999 端口无凭据保护，仅限局域网临时使用，不要长时间开放
- 接收端 curl 默认用 PowerShell curl（Invoke-WebRequest），用 `-o` 而非 `-O`

### 4.4 科学报告书写范式 🆕 v2.0

**Phase 报告应包含的 7 个节**（从 DENSITY_001_PHASE1_REPORT.md 提炼）：

1. **实验概述** — 核心命题 + 设计概要（一句话说清楚我们在干什么）
2. **数据汇总** — 采集规模 + 关键统计量（不要说过程，说结果）
3. **核心发现** — 每个发现独立一节，含统计证据 + 回归系数/显著度 + 原始引用
4. **假设检验** — 与假设树对齐的逐条验证表格
5. **方法论启示** — 不只是发现了什么，还有"这个方法本身教会了我们什么"
6. **通往下一阶段的路线** — 4 层递进，明确当前在哪一层
7. **数据与代码** — 关键文件清单 + 运行指标

**书写原则**：
- 报告面向"不了解本实验的人"，不是给自己的备忘录
- 用具体数字而非形容词（"88.5% 的残差为负"而非"大多数残差为负"）
- 每个发现必须有"证据强度"标注（强/中/弱/待验证）
- 区分"数据告诉我们的事实"和"我们对事实的解释"

## 五、Cron 驱动的长期自动化

### 5.1 设计原则

- **不写守护进程**：用 gateway cron 调度，不写 while true 循环
- **最小状态传递**：每次运行从文件读状态，写完退场
- **容错优先**：数据源不可达时不崩溃，标记为跳过后继续
- **渐进累积**：个别轮次无增量也要记录"本轮无有效增量"
- **模型无状态部署**：cron 触发 → 读取当前模型 JSON → 运行 → 写入结果。模型更新 = 替换 JSON 文件，无需重启 cron。

### 5.2 prompt 设计

- 告诉 agent 它是什么角色（"你是信息密度裁决者"）
- 列出所有核心文件路径
- 分步骤写清楚（1→2→3→...→验证→写入）
- 包含异常处理逻辑（"失败则跳过不阻塞"）
- 末尾放核心原则（"残差长期累积，不跳过记录"）

### 5.3 数据回证延迟

**问题**：数据需要未来信息才能标注（"这个信息后来有用吗？"）

**方案**：W=3 窗口回证
- 新记录标记 `validation_status: "pending"`，`validation_deadline: N+3`
- 每轮扫描 pending 块，检查是否有引用 → 回填 u 值
- 越 W=3 仍未回证的 → 标记为 u=0（视为无用）

### 5.4 数据处理韧性 🆕 v2.0

**DENSITY-001 暴露的真实问题及其解法**：

| 问题 | 根因 | 修复 |
|------|------|------|
| JSON 行截断（第98行） | cron 运行被超时 kill，写入未完成 | `train_and_deploy.py` 加 try/except per-line |
| 两个 JSON 挤一行（第20行） | cron 追加逻辑无 newline guard | 写入逻辑确保每条记录后跟 `\n` |
| Windows GBK 编码崩溃 | Python open() 默认用系统编码（中文Windows=GBK），JSON 含 emoji 无法编码 | `errors="replace"` 或 `$env:PYTHONIOENCODING='utf-8'` |

**核心教训**：在 Windows 上做 Python 数据管道，编码问题是排名第一的静默杀手。不像 Linux 默认 UTF-8，Windows 的 GBK 默认会在任何含 emoji 的 JSON 行上崩溃。

## 六、外部评审的价值

### 6.1 什么时候值得找人评审

- 方案落地后、正式跑起来之前
- 方法论的立场与实现存在张力时
- 需要不同模型/不同视角的独立判断

### 6.2 评审的黄金洞见

**问题不是"设计错了"，而是"设计中有自相矛盾"**：
- 我们说"不先验定义"→ 但 S·λ·R·C 就是先验框架
- 我们说"让模式涌现"→ 但 u 值的 1.0/0.5/0.3/0.0/-0.5 就是先验价值判断

**对策不是推翻重来，而是并行**：
- 保留先验框架（它有价值）
- 在旁边加一条真正的零先验轨道
- 让两条轨道的结论互相校准

### 6.3 评审意见的处理

```
收到评审 → 提取核心洞见 → 判断是否改变方法论 → 
是 → 修改方案（标注版本+外部评审驱动）→ 
保留原始评审文件作为独立文档 → 落盘变更记录
```

## 七、大语言模型的工程边界

### 7.1 能做的

- 方案设计、数学推导、代码生成
- 文档体系构建、格式统一
- prompt 编写、自动化流程设计
- 逻辑一致性检查、交叉验证

### 7.2 不能擅动的（须用户确认）

- 杀进程 (`taskkill` / `Stop-Process`)
- 删文件（不限 `del` / `rm`）
- 改系统配置（注册表、服务、防火墙）
- 执行需要管理员权限的操作

### 7.3 有效的工作模式

```
用户提出需求 → AI 设计方案 → AI 执行（在安全边界内）→ 
遇到边界外操作 → 精确告知用户需要什么操作（给具体命令）→ 
用户执行 → AI 继续
```

**关键**：不给抽象的"你应该配置一下XXX"，给**精确的可执行命令**。

### 7.4 三系统全自主协作模式 🆕 v2.0

**架构**：
```
雪飞提需求 → 闻声（OpenClaw）拆解调度 → 千手（Hermes）推演方案 → 
净瓶（Claude Code）实现 → 闻声交付
```

**全自主决策原则**：
- 不参与中间决策——只关注理想结果
- 遇到不明确需求/不可逆操作/成本超常/需求矛盾时才停下来问
- 日常裁决、回证、模型训练、报告生成全部自主

**实现方式**：
- Cron 驱动持续运行（judge cron 每6h，报告 cron 每天）
- 模型无状态热部署（替换 JSON → 下次 cron 自动读取）
- 异常自动检测但不中断（跳过坏块继续）

### 8.1 三层记忆体系

| 层 | 文件 | 内容 | 维护者 |
|---|------|------|--------|

1. 看最近几天日档
2. 提取可复用经验
3. 更新 MEMORY.md

### 9.2 合并旧仓库时的注意事项

**场景**：原仓库已存在（如 `main` 分支有旧文件），新 push 在 `master` 分支。

**操作**：
```bash
git fetch origin
git merge origin/main --allow-unrelated-histories
# 解决冲突
git commit
git push origin master
```

### 9.3 .gitignore 关键排除清单

```
# 个人记忆（安全）
memory/

# 临时文件

# 心跳管理
```

### 9.4 教训

- **推送前确认分支名**：旧仓库可能用 `main`，新 init 默认 `master`。直接 push 会创建第二个分支而非更新已有分支。
- **首次 push 后查浏览器**：确认文件列表是否正确，README 是否正确渲染
- **网络容错**：github.com 443 可能被墙/代理波动，做好重试准备

## 十、DENSITY-001 vs autoresearch 方法论对比 🆕 v2.0

### 10.1 结构同构

| 维度 | autoresearch (Karpathy) | DENSITY-001 |
|------|------------------------|-------------|
| 目标函数 | `val_bpb`（越低越好）| 残差 e=u-ρ（越接近零越好）|
| 迭代单元 | 5 分钟训练 | ~6 小时裁决 |
| 自主循环 | program.md → 改 train.py → 跑 → 看结果 → 保留/回滚 | cron → 裁决 → 回证 → 记录残差 → 下次使用 |
| 记忆系统 | Git（commit/回滚）| residual-series.jsonl + density-log.jsonl |
| 编排方式 | Markdown 文件 = 研究纲领 | cron prompt + 实验设计文档 |
| 框架依赖 | 零框架 | 零框架 |
| 代码规模 | 3 个核心文件 | 3 个核心 Python 脚本 |

### 10.2 关键差异

**autoresearch 的优势**：
- 目标函数完全客观（val_bpb 是硬件跑出来的数字）
- 反馈循环极短（5分钟），数据累积速度极快（一夜 100 个实验）
- 搜索空间可无限扩大（AI 可以改任何架构、超参数）

**DENSITY-001 的优势**：
- **元层**：不仅优化目标函数，还在验证测量工具本身是否准确
- **双轨**：先验 + 涌现并行，不会陷入单一框架的盲区
- **解释性**：Ridge 线性模型提供系数级解释（而非黑箱神经网络）

**autoresearch 的警醒**：
- "编排框架就是 LLM 本身"——读指令、理解指令、执行计划，不是 DAG
- "约束产生创造力"——AI 只能编辑一个文件，防止行动空间指数膨胀
- "永远不要停下来"——program.md 里的核心指令

### 10.3 对 DENSITY-001 的启示

1. 可以考虑在 Phase 3 引入"AI 自主修改 density_corrector.py 的架构"——从"人写模型 AI 跑数据"升级为"人写 program.md AI 迭代模型"
2. 减少人类参与的裁决环节，让 judge cron 更自主
3. 固定迭代预算（类似 autoresearch 的 5 分钟）可以保证实验进度可预期

## 十一、自主裁决系统设计规范 🆕 v2.0

### 11.1 静默灾难诊断

**定义**：阈值过高导致有效采样率趋近于零，系统在"看似正常运行"中失去诊断能力。

**DENSITY-001 的实例**：
- θ=1.00，32 轮中 31 轮 r_t=0
- 因为阈值太高，几乎没有块被保留，导致 "retention rate" 统计失去全部方差
- **诊断方法**：检查 retention rate 的时间序列，如果 >80% 的轮次 r_t=0，阈值过高

**对策**：
- θ 从 1.00 降至 0.90
- 设回滚条件：ē_W 连续 3 轮 < -1.0 → 回退至 0.95

### 11.2 Guardian 退役效应

**现象**：长期运行的自动检查模块（Guardian）退役时，单一事件产生极端残差（e=-2.639），拉低加权均值约 0.2。

**启示**：
- 单一高影响力事件的残差不应被当作系统漂移——它是点冲击
- 裁决系统应区分"点冲击"（单块极端残差）和"趋势漂移"（多轮 ē_W 方向一致移动）
- 解码权重（decision_enc）的系数 -0.202 表明：被解码的块在退役后无任何价值——它们在退役瞬间从"有用"变成"噪声"

## 十二、制度空间搜索 vs 参数空间搜索 🆕 v3.0

### 12.1 核心命题：端点星做的比 Karpathy 更难

Karpathy 的自动科研实验在**参数空间**搜索——优化模型权重，loss function 明确，反馈秒级。端点星的自进化系统在**制度空间**搜索——优化行为模组配置（钢印、基因模组），反馈信号稀疏，延迟天级，无 ground truth。

三个根本性困难：

1. **反馈信号稀疏** — 模型训练有 loss function，钢印效果只能数"阻止了几次事故"。一条钢印可能 30 天才触发一次
2. **延迟更长** — 不是秒级反馈，是天级甚至月级
3. **无 ground truth** — "这条钢印该不该降级"没有标准答案

### 12.2 DENSITY-001 的定位：给制度搜索装仪表盘

DENSITY-001 做的不是 Karpathy 式的参数搜索，而是**给第二种闭环装仪表盘**——用 ρ 量化"这个系统在变聪明还是变笨"。

当 Ridge 校正后的 ρ̂ 开始呈现稳定趋势（无论升降），端点星的信息密度就有了可量化的"体温计"。这让"系统在退化"从直觉变成数据。

### 12.3 三个新追踪指标的诞生逻辑

基于方法论对比，发现实验缺少 Karpathy 实验中自然具备的指标维度：

| 层次 | Karpathy 指标 | 端点星等价指标 | 状态 |
|------|-------------|--------------|------|
| 实验吞吐 | experiments/min | 裁决轮次/天 | ✅ 已有 |
| 信号质量 | loss/reward | ρ̄ 滑动均值 | ✅ 已有 |
| 评估精度 | validation accuracy | 残差 ē_W | ✅ 已有 |
| 行为多样性 | exploration entropy | BDI（新增）| 🆕 |
| 因果链深度 | gradient path length | RCD（新增）| 🆕 |
| 信息新鲜度 | data staleness | SDE（新增）| 🆕 |

**BDI（行为多样性指数）**：Shannon 熵衡量每轮 action_type 分布的多样性。BDI 下降 = 系统趋同化 = 可能陷入局部最优。

**RCD（引用链深度）**：引用有向图最长路径。RCD 增长 = 级联失效风险上升；RCD 趋零 = 系统碎片化。

**SDE（S 衰减有效性）**：P(referenced | λ<0.5) / P(referenced | λ≥0.5)。SDE≈1.0 = 衰减模型正确；>>1 = 衰减过激；<<1 = 衰减不足。

### 12.4 经验教训

1. **对比是最好的镜鉴** — Karpathy 的框架帮我们发现了三个盲区（行为多样性、因果链深度、信息新鲜度）。没有外部参照，系统会对自己缺少什么毫不知情
2. **指标不是越多越好** — 新增三个指标都有明确的判定逻辑和异常信号，不是为了"全面"而是为了覆盖已知盲区
3. **"阈值由数据涌现"不仅适用于 θ** — 三个新指标的异常阈值也应该从数据中涌现，不预设
4. **ρ̂ 趋势 > 单次 ρ̂ 值** — 体温计的价值不在于今天的体温，在于体温的趋势
