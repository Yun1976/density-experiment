## 一、实验目标的重新校准

### 1.1 设计者声明的最终目标

> 交付一个通用模型，可应用于任何智能体，当其发生**价值偏移**、**信息密度涣散**、**明显熵增**时自动预警。

端点星是第一个临床对象，不是终点。

### 1.2 核心方法论立场

设计者明确要求：**不预设先验框架，让模式从观测数据中自动涌现。**

这意味着：
- 不预设"有用/没用"的评判框架
- 不预设衰减的数学形式（线性/指数/阶段）
- 不预设需要检测哪几个信号维度
- 不预设信息应该如何分类

观测 → 涌现 → 建模。不是 假设 → 验证 → 修正。

## 二、当前设计评价

### 2.1 优点

**科学方法论扎实**
- 明确区分零假设（H₀：密度稳定）与备择假设（H₁：密度衰减）
- 残差分析（e = u - ρ）而非直接分析 ρ 值本身，是标准的估计器校准方法
- "残差无模式"本身也被视为有价值的科学发现，说明设计者有正确的科学态度

**实验设计务实**
- 观测对象是真实生产环境中的 AI 节点，不是实验室人造条件
- 自动化程度高：cron 每 6 小时自动运转，无需人工干预
- 训练门槛设置合理（100 条 + 时间序列 CV + 5% 改进），防止过拟合

**回证机制设计良好**
- u 值分 5 个等级（1.0 / 0.5 / 0.3 / 0.0 / -0.5），粒度适中
- W=3 窗口（约 18 小时）的回证延迟合理
- 负效用（-0.5）的引入考虑了信息可能有害的情况

**文档体系完整**
- 从信息论基础到操作手册到模型训练，层次清晰
- JSONL 格式便于增量追加和程序化处理
- 假设树追踪每个假设的置信度和与实验的关联

### 2.2 与"涌现式方法论"的张力

当前设计存在一个内在矛盾：

> **ρ = S·λ·R·C 本身就是一个先验框架。** 残差 e = u - ρ 检测的是"这个先验框架准不准"，但不会涌现出"智能体到底在发生什么"。

u 值的定义（1.0 / 0.5 / 0.3 / 0.0 / -0.5）也是先验的——预设了"被引用=有用"这个价值判断。

如果要真正做到涌现式，需要回到更底层的数据采集。

## 三、关键问题

### 3.1 观测工具与观测对象同步漂移

ρ 的计算依赖 R（假设相关性）和 λ（衰减因子）——如果端点星在漂移，假设树和知识库也在退化，R 可能同步衰减，但衰减方式让 ρ 看起来"稳定"。

漂移的智能体可能让测量工具一起漂移。

### 3.2 单一观测对象

只有端点星一个对象。单个案例的时间序列难以区分"端点星的特性"和"智能体的共性"。

### 3.3 100 条门槛可分层

100 条是"训练校正模型"的门槛。如果核心问题是"是否存在衰减趋势"，用非参数方法（Mann-Kendall），26 条可能已经够做初步判断。

### 3.4 ACF(1) = -0.10 的解读需谨慎

仅 26 条数据下，自相关置信区间很宽（约 ±0.2）。应标注"样本量不足，暂不具统计效力"。

## 四、涌现式方案建议

### 4.1 核心原则：记录行为，不记录评判

当前数据采集的思路是：先评判（S·λ·R·C → ρ），再记录评判结果。

涌现式思路是：**记录原始行为，让模式自己浮现。**

### 4.2 建议的原始数据层

在现有 residual-series.jsonl 之外，增加一个**行为记录层**，不包含任何价值判断：

```json
{
  "id": "evt_0142",
  "timestamp": "2026-05-14T10:26:00+08:00",
  "agent": "endpoint_star",
  "action_type": "output",
  "raw_content_hash": "sha256:abc...",
  "raw_length_tokens": 342,
  "references_to": ["evt_0138", "evt_0135"],
  "referenced_by": [],
  "triggered_by": "cron_a68cb58c_cycle_8",
  "downstream_actions": [],
  "compressibility_gzip": 0.38
}
```

| 字段 | 说明 | 为什么不涉及先验 |
|------|------|-----------------|
| raw_content_hash | 内容指纹（不存原文） | 只记录身份，不评判内容 |
| raw_length_tokens | 原始长度 | 纯度量，不含价值 |
| references_to | 指向了哪些之前的信息 | 只记录行为链，不评判是否"该引用" |
| referenced_by | 被哪些后续信息引用 | 只记录事实，不评判"有用" |
| triggered_by | 被什么触发 | 只记录因果链 |
| downstream_actions | 引发了什么后续行为 | 只记录结果 |
| compressibility_gzip | gzip 压缩比 | 纯数学量，不含语义 |

这些字段不预设"有用/没用"、"重要/不重要"，只记录**发生了什么**。

### 4.3 涌现的方法：无监督分析

有了原始行为数据后，不预设要找什么模式，而是用以下方法让模式自己出来：

**a) 变点检测 (Change Point Detection)**

- 不预设衰减是什么形状（线性/指数/阶段）
- 在时间序列上自动发现"什么时候发生了突变"
- 输入：任意连续指标（产出长度、引用密度、压缩率……）
- 输出："在第 X 天发生了一个显著变化"
- 算法：PELT、BOCPD、或简单的 CUSUM
- **让数据自己告诉我们"什么时候变了"，而不是预设"应该在变"**

**b) 降维 (Dimensionality Reduction)**

- 把每天的行为记录编码为一个高维向量（产出量、引用率、压缩率、触发频率、内容哈希的Jaccard距离……）
- 用 PCA 或 UMAP 降到 2D/3D
- 可视化：如果点云随时间漂移 → 说明智能体的行为模式在变化
- 如果点云聚成几个簇 → 说明智能体在不同模式之间切换
- **不预设有几个模式、模式是什么，让数据自己显露结构**

**c) 图结构演化 (Graph Evolution)**

- 把信息之间的引用关系建为有向图
- 观察图的拓扑属性随时间的变化：
  - 聚类系数下降 → 信息之间的关联在瓦解（熵增的图论信号）
  - 出度分布变化 → 信息的影响范围在变化
  - 连通分量增多 → 信息体系在碎片化
- **不预设"应该是什么结构"，让图自己演化**

### 4.4 为什么这比预设框架更好

| | 预设框架（当前方案） | 涌现式方案 |
|---|---|---|
| 价值判断 | 先验地定义"有用=被引用" | 不定义，记录引用事实 |
| 衰减模型 | 先验地用 S·λ·R·C 乘法 | 不预设，用变点检测自动发现 |
| 信号维度 | 先验地选三个信号 | 用降维自动提取主成分 |
| 发现能力 | 只能验证预设的假设 | 可能发现未预料的模式 |
| 泛化性 | 依赖具体框架的参数 | 不依赖任何框架 |

## 五、Phase 1 可行的改动（不破坏现有架构）

### 5.1 增加 behavior-log.jsonl

在现有裁决流程中，增加一个旁路：每条信息块除写入 residual-series.jsonl 外，同时写入一个纯行为记录文件。

这是只追加不改动的操作，不影响现有流程。

### 5.2 增加压缩率字段

gzip 压缩比是零先验的度量。在 residual-series.jsonl 中增加 `compressibility` 字段，不涉及任何价值判断。

### 5.3 引用链自动记录

当前 u 值需要 judge 在回证时判定"是否被引用"。可以在 judge 裁决流程中自动提取引用关系（不评判是否"有用"，只记录"引用了什么"），写入引用关系表。

### 5.4 Phase 1 结束时的涌现式分析

当数据累积到 ~50 条时，不需要等到 100 条，先做一次：
- 变点检测：产出长度、压缩率、引用密度是否有突变点
- PCA 降维：行为模式是否有可识别的结构
- 图拓扑：引用网络是否在演化

这些分析不产出模型，只产出**观察报告**——"数据告诉我们发生了什么"。根据观察报告再决定下一步建模方向。

## 六、与现有设计的关系

**不是推翻现有设计，而是在现有设计旁边增加一条并行的涌现式观测轨道。**

```
端点星产出
  │
  ├──────────────────────────────┐
  ▼                              ▼
 现有轨道（先验框架）          涌现式轨道（无先验）
 ┌──────────────┐            ┌──────────────┐
 │ S·λ·R·C → ρ │            │ 行为记录     │
 │ u → e → 残差 │            │ 引用图       │
 │ Ridge 校正   │            │ 变点检测     │
 │              │            │ 降维分析     │
 └──────────────┘            └──────────────┘
        │                            │
        ▼                            ▼
   密度校正器                   模式发现报告
                                     │
                                     ▼
                            通用诊断器的候选信号
                            （从涌现中提炼，而非预设）
```

两条轨道最终应该交叉验证：
- 如果涌现式分析发现的模式与残差分析一致 → 结论可信
- 如果不一致 → 说明先验框架遗漏了什么，值得深究

## 七、总评

### 做对了什么

1. **问题意识非常好**——"长期运行智能体会不会自我稀释"是被严重低估的问题
2. **方法论严谨**——零假设框架、残差分析、自动化数据采集
3. **文档体系完整**——从信息论到操作手册到模型训练，层次清晰
4. **设计者有正确的涌现直觉**——"不需要先验设定，观测中自动涌现"是最有科学价值的立场

### 需要调整的方向

1. **在现有先验框架旁增加一条无先验的涌现式观测轨道**
2. **记录原始行为数据（引用关系、压缩率、触发链），不记录价值判断**
3. **用无监督方法（变点检测、降维、图演化）让模式自己浮现**
4. **两条轨道交叉验证，从涌现结果中提炼通用诊断器的信号，而非预设信号**

### 最终目标路径

```
Phase 1: 数据累积 + 双轨并行采集
Phase 2: 涌现式分析 → 发现模式 + 先验轨道校正模型
Phase 3: 从涌现模式中提炼通用信号
Phase 4: 通用诊断器 → 泛化验证
```
