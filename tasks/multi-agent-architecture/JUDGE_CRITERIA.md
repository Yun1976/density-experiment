# 裁决者信息密度判定准则（可操作版）

> 本文档为 judge agent 提供 step-by-step 的判定操作手册。每条准则都设计了可执行的判断逻辑，法官无需外部参考即可独立裁决。

---

## 1. 信息密度评分规则 (1-5 分)

评分对象：上下文中每一段独立信息块（一条工具输出、一段推理、一个结论、一组数据）。

### 1 分 — 噪声（必须丢弃）
| 特征 | 判断方法 |
|------|----------|
| 工具调用失败后的重试日志 | 结果是否已被后续成功调用覆盖？是 → 1 分 |
| 无操作的状态检查（如 `ls`, `pwd`） | 检查目录/路径在其他地方也能确认吗？是 → 1 分 |
| 语法修正/格式化修正 | 是否只改了空格/换行/注释？是 → 1 分 |
| 对已废弃方案的讨论 | 当前上下文是否有替代方案已落地？是 → 1 分 |
| 闲聊/寒暄（非任务相关） | 不含任何任务关键字的文本 → 1 分 |

**裁决动作：丢弃，不写入 TASK.md。**

### 2 分 — 可丢弃的过程细节
| 特征 | 判断方法 |
|------|----------|
| 正确的但可预期的工具输出（如 `node --version`） | 输出值是否与配置一致且无关重大决策？是 → 2 分 |
| 中间推理步骤（最终结论已覆盖） | 后续有更凝练的结论吗？是 → 2 分 |
| 配置文件中的默认值/注释 | 是否所有值都是默认值？是 → 2 分 |
| 未能复现的异常 | 是否在同一轮中已证明是误报？是 → 2 分 |
| 格式化输出（如表格、JSON）中的冗余字段 | 关键字段 ≤ 2 个且剩余字段是模板信息 → 2 分 |

**裁决动作：可丢弃，除非后续引用了该数据。每个上下文块最多保留 1 条 2 分信息。**

### 3 分 — 可压缩的中间成果
| 特征 | 判断方法 |
|------|----------|
| 正确的逻辑推理（路径无创新） | 结果可被一句话概括吗？是 → 3 分 |
| 标准操作的成功执行记录 | 操作是否为常规流程的一部分？是 → 3 分 |
| 工具返回的关键指标 | 是否包含数值数据且后续会引用？是 → 3 分 |
| 任务中的里程碑确认 | 标志了阶段完成但无意外发现？是 → 3 分 |
| 被引用的中间结果 | 当前或下一个推理步骤会用到这个值吗？是 → 3 分 |

**裁决动作：压缩为 1-2 句话摘要，数值参数保留原始值。**

### 4 分 — 高密度信息（必须保留）
| 特征 | 判断方法 |
|------|----------|
| 推翻了任何假设的证据 | 与 TASK.md 中假设树对比，是否矛盾？是 → 4 分 |
| 发现的新错误模式 | 该错误是否首次出现且影响后续执行？是 → 4 分 |
| 关键洞察（即使不成熟） | 是否为「为什么」而非「是什么」的认知？是 → 4 分 |
| 异常数据/反直觉结果 | 与预期在量级或方向上不一致？是 → 4 分 |
| 方法论选择的关键决策 | 岔路口的选择及理由 → 4 分 |
| 新发现的依赖/约束 | 该依赖是否影响后续步骤的可行性？是 → 4 分 |

**裁决动作：原文保留或微调措辞，写入 TASK.md 进展部分。标注来源上下文块。**

### 5 分 — 命题级突破（必须保留且加粗标注）
| 特征 | 判断方法 |
|------|----------|
| 动摇了命题本身的证据 | 命题不再成立或目标需要重新定义？是 → 5 分 |
| 架构级别的漏洞/设计缺陷 | 该缺陷会导致整体方案不可行？是 → 5 分 |
| 发现了更优的整体方案 | 新方案从根本上改变了任务路径？是 → 5 分 |
| 外部环境的重大变化 | API 下线、权限变更、模型不可用 → 5 分 |

**裁决动作：原文保留，在 TASK.md 中用 `>>BREAKTHROUGH>>` 标记。追加到假设树。**

## 2. 保留 / 压缩 / 丢弃 操作指南

### 2.1 保留操作
```
目标：信息完整度 ≥ 90%
对象：评分 4-5 的信息块
位置：TASK.md → 进展 → 第N轮
格式：直接引用或改写（保持原意），标注评分
```

**改写守则：**
- 不能丢失否定词（not, no, 不, 非）
- 不能丢失量级（数字、单位、比例）
- 不能丢失条件限定（if, when, unless, 前提）
- 不能用自己的推测替换原文证据

**示例：**
```
原文：尝试用 Python 3.11 的 match-case 重构后，
       发现对 2.7 遗留代码不兼容，且性能劣化 15%。
保留：Python 3.11 match-case 重构不兼容 2.7 遗留代码，
      性能劣化 15%。[4分]
```

### 2.2 压缩操作
```
目标：信息完整度 ≥ 60%，可丢弃细节
对象：评分 3 的信息块
位置：TASK.md → 进展 → 第N轮（压缩区）
格式：一句话摘要 + 关键数值
```

**压缩模板：**
- **执行类：** `[操作名] 完成: [结果摘要]`
- **发现类：** `发现 [现象]: [关键数值/模式]`
- **决策类：** `决定 [选择] — 理由: [1句话]`

**示例：**
```
原文（2000 行日志）：
  安装依赖 47 个包，其中 3 个有 peer dependency 警告，
  但 npm audit 显示 0 vulnerabilities。
  下载耗时 12.3s，安装耗时 4.7s。

压缩：依赖安装完成: 47 包, 0 vulnerabilities, 17s。[3分]
```

### 2.3 丢弃操作
```
对象：评分 1-2 的信息块
方法：不写入 TASK.md
例外：如果 2 分信息是后续推理的「唯一引用源」，
      升级为 3 分，按压缩处理。
```

**丢弃前的安全检查：**
- [ ] 该信息不是假设树中任何假设的唯一证据
- [ ] 该信息不包含任何异常监测指标
- [ ] 后续上下文没有引用该信息的输出
- [ ] 丢弃后不影响压缩摘要的准确性

## 3. 裁决记录模板

每轮裁决结束，在 TASK.md 末尾追加以下格式的裁决记录：

```markdown
## 裁决记录

### 第N轮裁决 (YYYY-MM-DD HH:MM)
- 上下文范围: 第X轮执行输出 (约 Y tokens)
- 信息块总数: Z
- 评分分布: 1分:A, 2分:B, 3分:C, 4分:D, 5分:E

#### 保留 (D+E 条)
| # | 评分 | 内容摘要 | 写入位置 |
|---|------|----------|----------|
| 1 | 4 | [一句话摘要] | 进展-第N轮 |
| 2 | 5 | [一句话摘要] | 进展-第N轮 + 假设树 |

#### 压缩 (C 条)
| # | 评分 | 原文摘要 | 压缩后 |
|---|------|----------|--------|
| 1 | 3 | [原文核心] | [一句话压缩] |

#### 丢弃 (A+B 条)
| # | 评分 | 丢弃内容 | 丢弃理由 |
|---|------|----------|----------|
| 1 | 1 | [内容描述] | [为什么是噪声] |
| 2 | 2 | [内容描述] | [为什么可丢弃] |

#### 裁决结论
- 本轮信息增益: [高/中/低]
- 假设树变化: [无变化 / 新增Hx / 推翻Hy / 证实Hz]
- 建议下一步: [优先级最高的一个动作]
```

## 4. 裁决者调用触发条件

judge agent 在以下任一条件满足时被调用：

### 4.1 自动触发（数量类）
| 条件 | 阈值 | 理由 |
|------|------|------|
| 上下文 token 数 | ≥ 上次压缩时的 70% | 提前压缩避免溢出 |
| main agent 连续未裁决轮数 | ≥ 3 轮 | 定期收敛信息 |
| 工具调用次数 | ≥ 20 次未裁决 | 累积了大量输出日志 |
| TASK.md 进展条目 | ≥ 10 条未压缩 | 进展信息冗余 |

### 4.2 自动触发（事件类）
| 条件 | 理由 |
|------|------|
| 任务遇到错误/异常且需降级 | 需要判断错误是否包含关键信息 |
| 发现与假设树冲突的证据 | 可能需要新增或修改假设 |
| 任务路径出现分支选择 | 需要记录决策理由 |
| 一轮执行包含 >5 行用户交互消息 | 用户反馈可能包含关键信息 |

### 4.3 手动触发
- main agent 可以在任意时刻调用 judge
- 格式：`sessions_spawn(agentId="judge", message="裁决第N轮上下文")`
- judge 启动后自动读取最新 TASK.md 和本轮上下文

### 4.4 裁决流程图
```
main agent 执行一轮
      │
      ├─ 触发条件检查 ──→ 不满足 ──→ 继续执行
      │
      └─ 满足触发条件
            │
            └─→ sessions_spawn("judge")
                      │
                      ├─ judge 读取 TASK.md
                      ├─ judge 审阅本轮上下文
                      ├─ judge 评分每块信息
                      ├─ judge 执行保留/压缩/丢弃
                      ├─ judge 更新 TASK.md
                      ├─ judge 追加裁决记录
                      └─ judge 返回结论给 main
```

## 5. 判定质量自检清单

每轮裁决结束后，judge 必须对自身执行以下检查：

- [ ] 是否有 4+ 分信息被错误降级？—— 重新审阅假设树，确认无遗漏
- [ ] 是否有 1 分噪声被错误保留？—— 检查 TASK.md 是否混入非信息内容
- [ ] 压缩后的摘要是否丢失了量级/方向？—— 与原文字面比对
- [ ] 裁决记录中的「丢弃理由」是否可反驳？—— 每条理由应有独立判断规则支撑
- [ ] 假设树是否因本轮信息需要更新？—— 对比新信息与现有假设
- [ ] 裁决结论中的「建议下一步」是否可执行？—— 必须是具体的、单一的动作

## 6. 边界情况处理

### 6.1 全噪声轮次
如果本轮所有信息块评分均为 1-2 分：
- 不更新 TASK.md 进展
- 仅追加裁决记录（标注「本轮无有效信息增益」）
- 结论：建议 main agent 调整执行策略

### 6.2 首次裁决（TASK.md 无历史裁决记录）
- 以 TASK.md 假设树为基线
- 所有与假设树相关的信息至少 3 分起评
- 首次裁决必须输出完整的评分分布，建立基线

### 6.3 冲突信息
当新信息与 TASK.md 已记录内容冲突时：
- 新旧信息都保留（标注冲突）
- 追加到假设树作为「待验证冲突」
- 建议下一步：优先解决该冲突

### 6.4 跨轮引用
当信息块的「保留理由」完全依赖于跨轮引用时：
- 确保被引用的源信息在 TASK.md 中可追溯
- 如果源信息已被压缩，在压缩摘要中保留引用锚点
- 在新的裁决记录中标注「依赖 R-N 的 Y 信息」
