> **实验编号**: DENSITY-001  
> **启动日期**: 2026-05-14  
> **预计周期**: 3-6 个月（可能延长）  
> **负责人**: 观自在·闻声（OpenClaw）  
> **观测对象**: 端点星（EndpointStar, 192.168.1.5）  
> **最终交付物**: 一个可用的密度校正模型

---

## 一、命题

**智能体长期运行后，信息密度必然衰减。**

目前已知：
- 端点星出现过膨胀型输出（lint 连续两天 ρ≈0）——衰减的定性证据已存在
- 端点星执行过自发反膨胀（26→3 压缩）——衰减的对抗机制已出现
- 裁决框架的 S·λ·R·C 工程估计器是 ρ 的近似，不是真实密度

目前未知：
- 衰减的速率是多少？
- 衰减是否存在可识别的阶段（缓慢→加速→崩塌）？
- 自发反膨胀是否真正有效？
- 当前 ρ 估计器的误差有多大？
- 是否存在一个"永久记忆"的相变边界——超过某个密度阈值后信息不再衰减？

**本实验的目标：用残差数列回答这些问题，并最终交付一个可用的密度校正模型。**

## 二、核心概念

### 2.1 信息密度 ρ

当前工程定义（来自 SHANNON_DENSITY.md）：

$$\rho(x) = S(x) \cdot \lambda(t) \cdot R(x, \mathcal{H}) \cdot C(x, \mathcal{C})$$

其中 S = 香农惊喜度，λ = 衰减因子，R = 活跃假设相关性，C = 因果新颖性。

这是 H₀（零假设估计器）。本实验不预设它是对是错——让残差数列自己揭示。

### 2.2 残差 e

$$e(x) = u(x) - \rho(x)$$

- u(x) = x 在后续轮次中的实际效用（由回证确定）
- ρ(x) = 裁决时估计的信息密度
- e > 0 → 低估（信息实际比估计更有价值）
- e < 0 → 高估（信息实际是噪声）

**残差不是误差。残差是端点星"未被当前框架捕获的隐式信息密度"。** 残差数列的时间演化 = 端点星信息密度衰减模式的数学签名。

### 2.3 密度校正模型

最终交付物。输入端点星信息块的特征向量，输出校正后的密度 ρ̂。

$$\hat{\rho}(x) = \rho(x) + \hat{e}(x)$$

其中 ê(x) 是从残差数列中学习到的校正项。模型在以下条件下启用：
- 累积 ≥ 100 条完整回证的残差样本
- 时间序列交叉验证显示改进 > 5%
- 模型仅在 |ê| > 0.1 时实际应用校正

## 三、实验架构

### 3.1 观测对象

**端点星** — 192.168.1.5，局域网内长期运行的 AI 节点。执行业务包括知识库巡检（lint）、文档维护、系统事件响应、与观自在联动。

选择端点星作为观测对象的理由：
- 它是在实际业务中运行，不是实验室条件
- 它的信息产出是持续的、真实的
- 它的衰减是自然发生的，不是实验诱导的
- 结果可以推广到任何长期运行的智能体

### 3.2 测量仪器

**judge agent cron（a68cb58c）** — 每 6 小时自动触发密度裁决。

裁决流程：
1. 从端点星采集新信息（dir_list lint 报告等；失败不阻塞）
2. 信息块分割 → S·λ·R·C 评分 → 保留/压缩/丢弃
3. 后悔检查（W=3 窗口）+ 冗余检查（K=3 窗口）
4. 阈值校准、假设树更新
5. **残差记录**：每个信息块写入 residual-series.jsonl
6. **回证扫描**：对 W=3 内 pending 块确定 u 值 → 计算 e
7. 追加汇总到 residual-summary.jsonl

### 3.3 数据管道

```
端点星产出（工具输出/系统事件/自我行为）
        │
        ▼
┌───────────────────────┐

│ ── 残差记录            │
│ ── 回证扫描            │

        │
        ▼
┌───────────────────────┐
│ residual-series.jsonl │  ← 逐块记录（科研级数据）
│ residual-summary.jsonl│  ← 累积统计
│ density-log.jsonl     │  ← 轮次汇总
│ TASK.md               │  ← 假设树 + 裁决记录
└───────┬───────────────┘
        │ 数据充足时
        ▼
┌───────────────────────┐
│ density_corrector.py  │  ← 模型训练
```

### 3.4 自动化运转

无需人工干预。裁决 cron 每 6 小时自动运行，残差自动累积。数据达到门槛后模型自动训练。

人工介入点：
- 雪飞询问时，汇报残差趋势和端点星健康
- 端点星连续 3 轮不可达 → 触发通知
- 7 天周报 / 30 天月报（待设立 cron）

## 四、残差数列

### 4.1 数据结构

**residual-series.jsonl** — 每条记录代表一个信息块从估计到回证的完整生命周期：

```json
{
  "id": "b08_A_dual_offline",
  "cycle": 8,
  "timestamp": "2026-05-14T10:26:00+08:00",
  "block_id": "A",
  "source": "端点星双node同时离线",
  "S": 2.32, "R": 0.70, "C": 0.40, "lambda": 1.0,
  "rho_estimated": 0.65,
  "decision": "compressed",
  "u_observed": null,
  "u_source": null,
  "u_round": null,
  "residual": null,
  "validation_status": "pending",
  "validation_deadline": 11
}
```

回证后：
```json
  "u_observed": 1.0,
  "u_source": "Cycle 9引用: 退化升级链确认",
  "u_round": 9,
  "residual": 0.35,
  "validation_status": "validated"
```

### 4.2 u 值定义

| u 值 | 含义 | 触发条件 |
|------|------|---------|
| 1.0 | 明确有用 | 被后续轮次明确引用/重复发现需要对应 |
| 0.5 | 部分有用 | 压缩摘要被引用（原始信息已丢弃但摘要存活） |
| 0.3 | 隐式依赖 | 路径确认、架构假设等间接依赖 |
| 0.0 | 无用 | W=3 窗口内从未被引用或依赖 |
| -0.5 | 负效用 | 保留导致错误推理（罕见） |

### 4.3 初始状态（Cycle 1-8 回填）

| 指标 | 值 |
|------|-----|
| 总记录 | 32 条 |
| 已验证 | 26 条 |
| 待回证 | 6 条 |
| 残差均值 ē | -0.393 |
| 残差标准差 σ_e | 0.94 |
| 自相关 ACF(1) | -0.10 |
| 高估比例 | 88.5% |
| 膨胀型输出 | 3 个 |

解读：目前估计器有轻微系统性高估倾向（ē 为负），但 ACF(1) ≈ 0 表明无漂移趋势，方差在可控范围。26 条数据不足以做出统计结论。

## 五、密度校正模型

### 5.1 模型规格

- **算法**: Ridge 回归（带 L2 正则化的线性模型）
- **特征**: S, R, C, λ, ρ, cycle, decision（7 维）
- **输出**: 预测残差 ê
- **依赖**: 零外部库（纯 Python math）
- **存储**: JSON 格式（便于 cron 加载）

### 5.2 训练条件

| 条件 | 门槛 | 说明 |
|------|------|------|
| 最小样本数 | 100 条 | 硬性最低门槛，未达到不生成模型 |
| 验证方式 | 时间序列 CV | 训练集 = Cycle 1..N-1，验证集 = Cycle N |
| 部署条件 | TS-CV 改进 > 5% 且 观测 ≥ 20 | 否则不部署 |
| 校正启用 | 仅当 \|ê\| > 0.1 | 避免小噪声校正 |

### 5.3 版本管理

- `model/density_corrector_current.json` — 当前部署版本
- `model/density_corrector_YYYYMMDD.json` — 历史版本归档
- 每次重新训练自动归档旧版本

## 七、实验阶段

### Phase 1: 数据累积（当前）

**周期**: 启动 → 首次模型训练（~7 天）

**活动**:
- judge cron 自动运行，每 6h 裁决 + 残差记录
- 残差数列从 26 条增长到 100+ 条
- 回证延迟：新记录需等 W=3 轮（~18h）后才能获得 u 值

**里程碑**: 100 条完整回证残差 → 触发首次模型训练

### Phase 2: 基线模型

**周期**: ~30 天

**活动**:
- 首次模型训练，评估 TS-CV 性能
- 模型部署到 judge cron（若可靠）
- 观察校正后的 ρ̂ 是否降低后悔率/冗余率
- 特征重要性分析 → 识别 ρ 估计器的主要误差来源

**里程碑**: TS-CV 改进 > 5% → 模型可靠

### Phase 3: 模式识别

**周期**: ~90 天

**活动**:
- 残差时间序列分析：趋势、周期、断点
- 端点星健康状态与残差的关联分析
- 衰减模式分类（缓慢线性？指数？阶段性？）
- "永久记忆"相变边界的探索性分析

**里程碑**: 衰减模式的数学描述 → 可发表的发现

### Phase 4: 泛化验证

**周期**: ~180 天

**活动**:
- 模型能否用于其他智能体（千手/净瓶）？
- 信息密度健康指数的设计
- 研究范式总结

**里程碑**: 泛化验证通过 → 工具化交付
