# 数据格式

本项目**不含原始数据**(原始笔记含私有知识库内容)。这里只说明格式,你要自行构造同格式数据。

## 格式:JSONL,每行一条

```json
{"instruction": "<评估器 prompt>", "input": "<笔记全文>", "output": "<1-5>", "filename": "<笔记标识>"}
```

| 字段 | 说明 |
|---|---|
| `instruction` | 评估器 system prompt(定义评分标准和原则,所有样本共用同一段) |
| `input` | 笔记全文(建议截断到 3000 字以内) |
| `output` | 人工评分 1–5(字符串) |
| `filename` | 笔记唯一标识(文件名/ID,用于结果对照) |

见 [`sample.jsonl`](sample.jsonl) 的 3 条 fake 示例。

## 评分定义

| 分数 | 含义 |
|---|---|
| 1 | 可删流水(日报/草稿) |
| 2 | 可合并归档 |
| 3 | 一般参考 |
| 4 | 压缩保留 |
| 5 | 必留核心(被反复引用) |

## 如何构造你自己的数据

1. 从你的知识库(笔记/Obsidian/文档库)收集笔记全文
2. 人工评分 1–5(建议多人评分取共识,核查一致性)
3. 拼成 `instruction/input/output/filename` 的 JSONL

## 数据质量建议(踩过的坑)

- **量**: 至少 500 条。本项目用 300 条,LLM 学不过 TF-IDF 回归。
- **类别平衡**: 每个分数至少 30–50 条。本项目分1 仅 2 条、分4 仅 13 条,模型学不会输出这两个分数。
- **去重**: 内容重复的笔记要去重。本项目原始数据 22 条重复(如多个目录的同名文件)。
- **标签一致性**: 同一篇多人/跨天评分不一致 = 噪声,是天花板。建议评分前先校准标准。
- **信号源**: 想清楚「存活价值」度量的是「内容质量」(信号在文本,LLM 有牌打)还是「被调用频次」(信号在 metadata,LLM 读全文看不到)。后者回归天然占优。
